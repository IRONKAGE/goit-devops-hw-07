# **Завдання 7: Вивчення Helm**

## **Опис завдання:**

Ваше завдання — створити кластер `Kubernetes` у тій самій мережі (`VPC`), яку ви налаштували в попередньому домашньому завданні, та реалізувати такі компоненти:

1. **Створення кластера Kubernetes** через `Terraform`
2. **Налаштування Elastic Container Registry (ECR)** для зберігання `Docker-образу` вашого `Django-застосунку`
3. **Завантаження Docker-образу Django** до ECR
4. **Створення helm chart** (`deployment.yaml`, `service.yaml`, `hpa.yaml`, `configmap.yaml`)
5. **Перенесення змінних середовища (env)** з теми 4 в `ConfigMap`, який буде використаний вашим застосунком

## **Кроки виконання завдання:**

1. **Створіть кластер `Kubernetes`:**
   - Використовуючи Terraform, створіть кластер Kubernetes у вже існуючій мережі (VPC)
   - Забезпечте доступ до кластера за допомогою kubectl

2. **Налаштуйте `ECR`:**
   - Використовуючи `Terraform`, створіть репозиторій в Amazon Elastic Container Registry (`ECR`)
   - Завантажте `Docker-образ Django`, який ви створювали в темі 4, до `ECR`, використовуючи `AWS CLI`

3. **Створіть `helm`. У Helm-чарті має бути реалізовано:**
   - **Deployment** — з образом Django з ECR та підключенням `ConfigMap` (через `envFrom`)
   - **Service** — типу `LoadBalancer` для зовнішнього доступу
   - **HPA (Horizontal Pod Autoscaler)** — масштабування подів від 2 до 6 при навантаженні > 70%
   - **ConfigMap** — для змінних середовища (перенесених із теми 4)
   - **values.yaml** — з параметрами образу, сервісу, конфігурації та autoscaler

## **Бонус-завдання: для тих, хто має власний домен**

> ❗За це завдання НЕ передбачені додаткові бали, проте ви зможете отримати фідбек від ментора та відпрацювати практичні навички.

> Якщо ви купили домен (або маєте доступ до нього), можете додати в Helm-чарт підтримку **Ingress** + **TLS** як **бонусне завдання**:

### **Задача:**

1. **Додайте Ingress-ресурс** до Helm-чарта, який буде вмикатись через `values.yaml`:
   - Можливість задання домену (`host`)
   - Підтримка HTTPS (`tls: true`)
   - Вказівка ingressClassName (наприклад, `nginx`)
2. **Підтримка cert-manager**:
   - Додайте анотацію `cert-manager.io/cluster-issuer: letsencrypt-prod` (або test)
   - Очікується, що студент самостійно встановить `cert-manager` через Helm (можна за готовою документацією або з минулих занять)

***Приклад параметрів у `values.yaml`:***

```yaml
ingress:
  enabled: true
  className: nginx
  host: yourdomain.com
  path: /
  pathType: Prefix
  tls: true
```

**Результати виконання:**

   - У вашому AWS-акаунті створено кластер `Kubernetes`
   - `ECR` містить завантажений Docker-образ Django-застосунку
   - Застосунок розгорнутий у кластері за допомогою `Helm`-чарта
   - `Service` забезпечує доступ до застосунку через публічну IP-адресу
   - `ConfigMap` підключено до застосунку через `Helm`
   - `HPA` динамічно масштабує кількість подів
   - ***(Бонус)*** Якщо є домен, налаштовано `Ingress` із `TLS` через `cert`-manager.

**Критерії прийняття завдання:**

1. Кластер `Kubernetes` створений через `Terraform` і працює
2. `ECR` створений і містить завантажений `Docker`-образ
3. `Deployment`, `Service` і `HPA` створені та працюють у кластері за допомогою `helm`
4. `ConfigMap` створено та використовується застосунком
5. Проект запушено у `GitHub`-репозиторій у гілку `lesson-7` з документацією в `README.md`

## **Опис архітектури (God Mode):**

Цей репозиторій демонструє розгортання production-ready інфраструктури з використанням сучасних Cloud-Native підходів:

1. **Infrastructure as Code:** Використання `Terragrunt` для DRY-архітектури та автоматичного створення S3-бакетів для стейту.
2. **Kubernetes (EKS):** Створення відмовостійкого кластера у кількох зонах доступності (Multi-AZ) з увімкненим логуванням (CloudWatch).
3. **Helm Charts:** Динамічне розгортання Django-застосунку з використанням `Deployment`, `Service` (ClusterIP) та `HPA` (Автомасштабування).
4. **Ingress & TLS:** Використання `ingress-nginx` та `cert-manager` для автоматичного отримання SSL-сертифікатів від Let's Encrypt.
5. **Security First:** Всі паролі та токени винесені у файл `.env`, контейнери працюють від непривілейованого користувача, а ECR сканує образи на вразливості.

## 🚀 Підготовка до запуску (Секрети)

Перед першим запуском вам необхідно налаштувати локальне середовище:

1. Створіть файл `.env` у корені проекту на базі `.env.example`:

   ```bash
   cp .env.example .env
   ```

2. Відкрийте `.env` і заповніть ваші реальні дані:
   - Ключі AWS (`AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`).
   - Паролі для бази даних та Django.
   - Токен LocalStack Pro (якщо тестуєте локально).

> ⚠️ **Важливо:** Файл `.env` додано до `.gitignore`. Він ніколи не потрапить у репозиторій. Файли `.tfvars` містять лише безпечні налаштування інфраструктури (регіони, типи інстансів) і безпечно зберігаються в Git.

## 🚀 Як користуватися (Автопілот)

### Опція А: Тестування локально (Через LocalStack)

*Ідеально для перевірки Terraform коду та збірки Docker-образів. Потребує LocalStack Pro для симуляції EKS.*

```bash
# Розгортання середовища dev локально
make deploy-local dev
```

### Опція Б: Деплой в реальний AWS (Бойовий запуск)

*Завдяки Terragrunt, проблема "курки та яйця" вирішена! Більше не потрібно вручну коментувати файли чи створювати S3-бакети. Terragrunt автоматично створить інфраструктуру для стейту перед запуском Terraform.*

**Базовий деплой (Без домену):**

Додаток буде розгорнуто, але доступ буде через технічну адресу балансувальника (Load Balancer).

```bash
make deploy-aws prod
```

**Повноцінний деплой (Із доменом та HTTPS):**

Скрипт автоматично встановить Ingress Controller, Cert-Manager, налаштує маршрутизацію та випише SSL-сертифікат для вашого домену.

```bash
# Формат: make deploy-aws [середовище] [ваш_домен]
make deploy-aws prod my-super-django-app.com
```

### 🌍 Середовища (Environments): `dev` vs `prod`

Наш код є адаптивним і керується через `make`.

🟢 **Середовище `dev`**

- **Архітектура:** Економна (t3.medium інстанси).
- **Мережа:** NAT Gateway вимкнений (для економії).
- **Призначення:** Швидка перевірка застосунку розробниками.

🔴 **Середовище `prod` (Бойовий режим)**

- **Архітектура:** Високонавантажена (`m5.large`, Multi-AZ для відмовостійкості).
- **Мережа:** Увімкнено NAT Gateway для безпечного доступу вузлів до інтернету.
- **Безпека:** Активне сканування Docker-образів в ECR та аудит-логи в EKS кластері.
